# 卡素民宿 AI 配置与交付规划 Agent｜最小改造指导

> 适用仓库：当前 `MoveRenovateAI` 项目  
> 目标：在不重构整套系统的前提下，将现有“搬家 + 装修规划 Agent”改造成可用于求职作品集演示的：
>
> **卡素民宿 AI 配置与交付规划 Agent**
>
> 项目定位：基于真实已建成“卡素民宿”案例沉淀的**回溯式 AI PoC**。  
> 展示“如果后续同类装配式民宿项目引入 AI，可如何辅助产品选型、配置建议与交付任务规划”。

---

- 卡素民宿是真实已完成的装配式民宿案例；

## 2. 改造目标与最小闭环

### 最终一句话

> 面向装配式民宿扩建或新建需求，用户输入面积、客群、运营方式与场地条件后，系统基于既有产品规格和示例规则，生成户型选型、空间配置、智能设备建议及交付任务计划。

### 必须跑通的闭环

```text
用户填写需求
  → 空间配置 Agent
  → 推荐 38㎡ / 56㎡产品与配置方案
  → 用户确认或调整
  → 交付规划 Agent
  → 生成阶段、任务、验收点与风险提示
  → 保存为项目档案
```

### 本次只做一个固定 Demo 案例

```text
案例名称：威海滨海度假区 38㎡单体民宿扩建（Demo）

场地类型：滨海度假区
客群：情侣 / 2人短住旅客
产品选择：Capsule Cabin S，38㎡单体
风格：柔和粉白色系、海滨轻度假
运营模式：自助入住 + 基础智能控制
交付方式：工厂预制 + 运输 + 现场吊装 + 调试验收
```

## 4. 原模块与新模块映射

| 现有模块 | 当前语义 | 改造后的语义 | 改造方式 |
|---|---|---|---|
| `Renovating Agent` | 装修需求、预算、施工阶段 | **空间配置 Agent** | 保留骨架，替换节点逻辑和 Prompt |
| `Moving Agent` | 物品、打包、搬运、还原 | **交付规划 Agent** | 保留 LangGraph 骨架，替换全部业务节点 |
| `projects` | 搬家/装修项目 | 民宿扩建/新建配置项目 | 保留并增加少量字段 |
| `checklists` | 搬家物品/装修清单 | 产品配置、交付准备、开业准备清单 | 改名称和数据 |
| `budgets` | 家装预算 | 配置预算区间 | 保留结构，不接真实单价 |
| `phases` | 家装施工阶段 | 设计确认—预制—运输—吊装—调试—验收 | 保留结构，改种子数据 |
| `conversations` | 搬家/装修对话 | 配置调整、需求澄清记录 | 直接复用 |
| `RAG` | 搬家/装修知识 | 产品规格、模块说明、交付注意事项 | 轻量替换默认知识 |

## 6. Agent 改造要求

### 6.1 空间配置 Agent

对应原文件：

```text
backend/app/agent/renovating_agent/
├── nodes.py
├── graph.py
└── __init__.py
```

建议对外改名为：

```text
空间配置 Agent
```

若改文件夹会导致大量导入问题，可保留目录名，仅修改类名、函数名、文案和业务逻辑。

#### 节点顺序

```text
1. 解析民宿需求
2. 推荐产品型号（38㎡ / 56㎡）
3. 生成空间模块与室内配置
4. 生成基础智能设备建议
5. 生成示例预算区间
6. 输出结构化方案
```

#### Agent 输入字段

| 字段 | 示例 |
|---|---|
| 面积偏好 | 38㎡ / 56㎡ / 不确定 |
| 客群 | 情侣、亲子、小团体 |
| 风格 | 卡素系列、自然原木、海滨轻度假 |
| 运营方式 | 自助入住、人工服务、长租 |
| 智能化等级 | 基础 / 标准 |
| 场地条件 | 滨海、交通条件、吊装空间、基础完成情况 |
| 预算等级 | 基础 / 中档，不输出真实金额 |

#### Agent 输出必须包含

```json
{
  "recommended_product": "Capsule Cabin S",
  "recommended_area_sqm": 38,
  "recommendation_reason": "",
  "space_modules": [],
  "configuration_items": [],
  "smart_devices": [],
  "budget_range": {
    "level": "中档",
    "note": "示例配置预算区间，仅用于 PoC 演示，不构成正式报价。"
  },
  "assumptions": [],
  "human_confirmation_required": []
}
```

### 6.2 交付规划 Agent

对应原文件：

```text
backend/app/agent/moving_agent/
├── nodes.py
├── graph.py
└── __init__.py
```

建议对外改名为：

```text
交付规划 Agent
```

若改名会影响范围，则保留目录，但对外名称必须显示为“交付规划 Agent”。

#### 节点顺序

```text
1. 读取已确认配置
2. 生成交付阶段
3. 生成每阶段任务与验收点
4. 生成现场风险提示
5. 输出交付看板数据
```

#### 统一阶段

```text
方案确认
→ 工厂预制与集成
→ 运输准备
→ 现场基础检查
→ 吊装与模块安装
→ 室内/智能设备调试
→ 验收与开业准备
```

#### 每阶段输出字段

```json
{
  "phase": "",
  "tasks": [],
  "acceptance_criteria": [],
  "risks": [],
  "human_confirmation_required": true
}
```

风险提示必须包含：

- 滨海环境防腐、防水；
- 现场基础与吊装条件；
- 运输尺寸与道路条件；
- 智能设备网络、供电与调试；
- 消防、结构、安全等需由专业人员确认。

## 7. 数据模型最小修改

尽量不重构数据库，只在现有 `Project` 中增加必要字段，或先放到已有 `requirements JSONB` 字段中。

### 推荐优先方案：先使用 `requirements` JSONB

```json
{
  "product_model": "Capsule Cabin S",
  "target_guests": "情侣 / 2人短住旅客",
  "operation_mode": "自助入住 + 基础智能控制",
  "intelligence_level": "基础",
  "site_conditions": ["滨海", "需确认吊装空间", "需确认基础条件"]
}
```

只有在前端展示确实不方便时，再新增字段：

```text
product_model
target_guests
operation_mode
intelligence_level
```

### 数据表语义转换

| 表名 | 新语义 |
|---|---|
| `projects` | 民宿扩建/新建配置项目 |
| `checklists` | 产品配置清单、交付准备清单、开业准备清单 |
| `checklist_items` | 外壳模块、门窗、卫浴、床具、茶吧、照明、门锁、传感器、网络预留等 |
| `budgets` | 模块化建筑、室内集成、家具软装、智能设备、运输吊装等示例预算类别 |
| `phases` | 7 个装配式交付阶段 |
| `conversations` | 需求调整与方案沟通记录 |
